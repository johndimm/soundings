Earprint Dev Diary

2026-04-01

Problem: Spotify rate limiting — stats show nothing

  • Hitting Spotify's 30-request/30s rate limit repeatedly. But the in-app call tracker showed zero calls at the time of the 429.
  • Root cause: the call tracker only measures calls through /api/next-song. All browser-side Spotify calls — SDK internal retries, polling, device reclaim — are invisible to it.
  • Three sources identified:
    1. SDK auth retry loop: on authentication_error the SDK kept retrying _refreshToken → GET /v1/melody/v1/check_scope in a tight loop.
    2. Polling: getCurrentState() every 1 second — 60 calls/minute direct to Spotify.
    3. visibilitychange reclaim: fired PUT /v1/me/player on every tab switch with no throttle.

Fix: Stop SDK retry loop

  • In authentication_error listener: call player.disconnect() before setting error state. This stops the SDK from spinning on failed token refresh.

Fix: Replace Spotify polling with local animation

  • Removed getCurrentState() polling entirely — was the largest source of invisible Spotify calls.
  • Replaced with a 250ms local setInterval that increments sliderRef by 250ms when not paused and not seeking. Zero Spotify API calls.
  • Auto-advance (fire at 98% of duration) moved into the local timer. Progress bar is smooth; player_state_changed events resync position on real state changes.
  • visibilitychange handler throttled to at most one reclaim every 10 seconds.

Fix: Crossfade between tracks

  • Added fadeVolume(player, from, to): 20 steps over 700ms using player.setVolume() (SDK-local call, not a Spotify Web API call — no rate limit concern).
  • advanceWithFade(): fades out current track, sets pendingFadeInRef, calls advance().
  • Play effect: when pendingFadeInRef is set, starts new track at volume 0 then fades in.

Fix: Rating hammering Spotify

  • Every rating triggered a full LLM + Spotify lookup cycle, hammering the rate limit on rapid slider adjustments.
  • Split into two phases:
    1. On rating: fetchProfileOnly() — calls /api/next-song with profileOnly: true. LLM runs, Spotify lookup skipped. Returns raw song suggestions + updated profile. Suggestions shown in "DJ is thinking…" panel below the player.
    2. When songs actually needed: resolvePending() — calls /api/next-song with songsToResolve: [...]. Skips LLM entirely, just resolves the suggestions to Spotify tracks. Triggered lazily by the fill effect when buffer is empty.

Fix: Profile text quality

  • The "What I know about you" profile was a structured coordinate log (LIKED: (42,28) warm soul | DISLIKED: ...) — more like a debug dump than a human description.
  • Updated the LLM system prompt: profile field is now 2-3 natural sentences describing the listener's taste in terms of genres, eras, moods, instruments, and energy. Under 60 words. "Write like a DJ's notes on their audience."

2026-03-30

Problem: 401 Unauthorized spam on Vercel

  • On Vercel, POST /api/next-song immediately returned 401 (Unauthorized).
  • The client had no backoff for this error — the auto-fill effect loop retried immediately and endlessly (hundreds of requests per session).
  • User saw "Could not load songs. Try again." with the browser network tab flooding with 401s.
  • Root cause #1: Infinite retry loop — any non-rate-limited error from fetchToBuffer set loadingQueue=false in .finally(), which re-triggered the auto-fill useEffect, which called fetchToBuffer() again. No backoff for generic errors.

Fix: Infinite retry loop

  • Added class AuthError extends Error for 401 responses specifically.
  • In fetchCards: throw AuthError when res.status === 401.
  • In fetchToBuffer catch: on AuthError, redirect to / (re-auth). On generic errors, apply the same 30-second backoff used for rate limiting.

Problem: Refresh token cookie missing on Vercel

  • The 401 from /api/next-song was caused by missing cookies: access token and/or refresh token not found server-side.
  • Vercel log: "refreshSpotifyAccessToken: no refresh token cookie found".
  • The user was on /player (SSR reads the access token cookie fine), but the API route couldn't find it.
  • Root cause #2: The OAuth callback (app/callback/route.ts) used cookies().set() from next/headers followed by redirect() from next/navigation. In Next.js 16, redirect() throws a special error — cookies set before the throw may not be flushed into the response headers. On Vercel this meant only the first cookie (access token) was reliably set; the expiry and refresh token cookies were dropped.

Attempts to fix cookie setting

  • Tried NextResponse.redirect() with response.cookies.set() — broke local login entirely.
  • Reverted; added logging to confirm Spotify returns a refresh token (it does).
  • Tried plain new Response(null, { status: 302, headers }) with explicit Set-Cookie headers — worked locally, still failed on Vercel.
  • Current attempt: return a 200 HTML response with Set-Cookie headers and window.location.replace('/player'). Avoids redirect entirely — Vercel's edge network may strip Set-Cookie from 3xx responses but should leave them on 200 responses.

Why it works locally but not on Vercel

  • Locally (http://localhost:3000): NODE_ENV=development, cookies have no Secure flag, responses flow directly from Next.js dev server to browser with no proxy.
  • On Vercel: NODE_ENV=production, cookies have Secure; SameSite=Lax. All traffic goes through Vercel's edge network proxy. The proxy may strip Set-Cookie headers from 302 redirect responses (either for security, caching, or CDN normalisation reasons). The 200 HTML approach bypasses this.

Status

  • Infinite retry loop: fixed (30-second backoff on all errors; immediate redirect to login on 401).
  • Cookie persistence on Vercel: in progress — testing 200 HTML response approach.
