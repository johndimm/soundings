Soundings Dev Diary

2026-04-19

"Load startup channels" button

  • After System Reset the persisted channel list is exactly one empty All row. The blank-slate auto-loader intentionally skips the server fetch in that case (allOnlyEmptyWipe short-circuit), so the user lands on a barren player with nothing to play and no obvious next step.
  • Fix: player now renders a Load startup channels button in the empty state (no current card, no queue, channels ≤ 1 and either empty or the empty All). Clicking it hits GET /api/startup-channels?source=spotify|youtube, which reads the user-curated bundles:
    • source=youtube → data/factory-channels-youtube.json
    • source=spotify → data/factory-channels.json
  • Response is parsed through parseChannelsImport, wrapped with ensureAllChannel, persisted via saveChannels, and the preferred active channel is hydrated through loadChannelIntoState. A missing file or invalid shape renders a small error under the button instead of silently failing.
  • The button is source-aware: if the user is in YouTube-only mode (or has Spotify picked in settings) the label copy and the file it fetches follow suit, so the Spotify bundle never leaks into a YouTube session and vice-versa.

Spotify login flicker loop (YouTube-test fixture leak)

  • Symptom: logout → pick Spotify → screen flickers; server log floods with POST /api/youtube-resolve-test 200.
  • Root cause: when youtubeResolveTestActive was true (which can linger in a dev server whose YOUTUBE_RESOLVE_TEST env was set before a .env.local edit without restart), the client sent youtubeResolveTest: true on every /api/next-song call — regardless of source. The server's profile-only branch then returned a YouTube fixture for every suggestion, even to Spotify callers. The fixture landed in the suggestion buffer, resolveOneSuggestion noticed it via isYoutubeResolveTestFixtureSuggestion(s) and hit /api/youtube-resolve-test, got a YouTube track, Spotify couldn't play it, the advance retried, and the cycle repeated.
  • Fix (client): renamed the effective flag to testModeForSource = youtubeResolveTestActive && sourceRef.current === 'youtube'. Used in fetchSuggestions (fixture short-circuit), the youtubeResolveTest body field on all three call sites (fetchSuggestions, fetchProfileOnly, resolveOneSuggestion), and the /api/youtube-resolve-test URL switch. Dropped the isYoutubeResolveTestFixtureSuggestion(s) OR condition — it was the amplifier that turned one server slip into a loop.
  • Fix (server): added rawSource === 'youtube' to the guard on /api/next-song's fixture short-circuit. Belt-and-suspenders so a misbehaving or legacy client can't trick the server into returning a YouTube fixture to a Spotify session.

YouTube: force start at 0s

  • Symptom: some tracks jumped straight to ~30s on first play, cutting off the intro.
  • Root cause: YouTube's "resume where you left off" feature remembers playback offsets for signed-in viewers and applies them even to embedded iframes, overriding our autoplay=1.
  • Fix (three layers):
    • Added &start=0 to the embed URL — declarative hint.
    • In onReady, unconditionally seekTo(0, true) before playVideo() — runs exactly once per mount (component is keyed by track id).
    • Safety net in onStateChange: on the first PLAYING transition, if getCurrentTime() > 1.5, seekTo(0) and log [yt] detected resume offset, snapping to 0. A one-shot ref (didSeekToZeroOnPlayRef) guarantees it fires at most once.
  • The YTPlayer TypeScript interface gained a seekTo(seconds, allowSeekAhead?) method to match the IFrame API.

Queue delete = 0.5★ skip

  • Clicking × on an Up Next row used to just drop the track silently. The DJ had no idea the user rejected it and would happily suggest the same sonic territory again.
  • Fix: removing a track from the queue now records a 0.5-star rating (lowest rung above zero — "skip this sound"), added to both cardHistory and sessionHistory. The auto-persist effect writes it to the channel on the next tick.
  • Guard: if the track is already in history (user previously rated it), we do not overwrite that rating — a thoughtful 4★ stays 4★ even if you later clear it from the queue.
  • New log: [dj-queue] queue-remove: recorded 0.5★ <track> — <artist>.

All channel learns from every channel

  • The All channel has always been special — no genre/region/artist/notes config, just discovery pinned to 100. Previously its DJ context was only its own sessionHistory / cardHistory, so moods developed on Jazz or Baroque never informed All.
  • Fix: when the active channel is earprint-all, the DJ now merges sessionHistory and cardHistory across every channel before calling the LLM. All's own refs take precedence (freshest), and other channels overlay dedup-by-track|artist. Applied to fetchSuggestions (alreadyHeard), fetchToBuffer (sentHistory), and fetchProfileOnly (sessionHistory + alreadyHeard).
  • Per-channel sampling keeps the payload small: each non-All channel contributes a stratified sample — last 15 recent + top 10 highest-rated + 5 lowest-rated (dedup happens in the merge map). Channels with ≤30 entries contribute everything; larger channels stay bounded. All's own history is sent in full (current-direction fidelity for the active channel).
  • Each channel keeps its own persisted history — All's merge is read-only at query time. Ratings you leave on Jazz contribute to All's taste profile on the next DJ turn without mutating Jazz's channel file.
  • New log: [dj-queue] All channel: merged histories from all channels {channels, sessionEvents, cardEntries, perChannelSample, contributions} fires on every DJ call while All is active.

Per-source factory channels

  • Factory channel snapshots are now source-specific. The save-factory API accepts ?source=spotify|youtube and writes data/factory-channels.<source>.json; the blank-slate fetch and "reload factory channels" reset both go through the same per-source path (with a shared factory-channels.json fallback, then the built-in import).
  • Added a dev-only "Save factory file" button in Settings that captures the current browser's channels as the factory default for whichever source is active. Distinct from the long-standing "Save current channels to server file" which still writes the shared snapshot.

Source switch clears per-channel playback, preserves user data

  • Flipping source (or logging in via a different path) now wipes currentCard, queue, playbackPositionMs, and playbackTrackUri on every channel — but keeps history, ratings, genres, notes, sliders, and the taste profile. This was the fix for "selected YouTube but Spotify is playing": leftover Spotify tracks in each channel's queue were still being dispatched to the Spotify SDK.

Source selector removed from Settings

  • The in-app "Playback Source" toggle was letting the user end up in a self-inconsistent state: source set to YouTube, localStorage still full of Spotify currentCard / queue entries, Spotify SDK still connected.
  • Removed the selector. Source is now implied purely by how the user signed in — Spotify OAuth callback appends ?spotify_login=1 and clears the YouTube cookie; /api/auth/youtube sets the YouTube cookie and appends ?youtube_login=1. Logout clears both. One source of truth.

Fresh-login localStorage scrub

  • Race condition: PersistentPlayerHost is wrapped in <Suspense> (for useSearchParams). On the server the fallback renders — which includes PlayerClient — and on the client that fallback hydrates first, letting PlayerClient read stale channel data from localStorage before the Suspense resolves and any parent-level reset could run.
  • Extracted the reset into app/lib/freshLogin.ts with a module-level "already applied" flag. Called from both PersistentPlayerHostInner's render body and the top of PlayerClient's hydration effect (before loadChannels()). Whichever fires first wins; the other is a no-op.
  • The scrub inspects each track's source field directly instead of trusting settings.source, so wrong-source entries get dropped even if the settings field already matches the new login.

YouTube playback: three separate failure modes fixed

  • Autoplay reliability. Chrome frequently blocks cross-origin iframe autoplay-with-sound for freshly mounted iframes. autoplay=1 in the embed URL was not enough on its own.
  • PlayerClient now imperatively calls youtubePlayerRef.current?.play() when currentCard changes to a YouTube track, letting the YoutubePlayer handle (YT JS API → postMessage fallback → pendingPlayRef latch for onReady) carry the user gesture forward.
  • postMessage origin mismatch. The fallback command path sent postMessage with target origin https://www.youtube.com. Chrome refuses to deliver when the iframe's contentWindow.location is still about:blank (freshly mounted) — silent failure, overlay stuck on top of a video that otherwise would have played.
  • Changed target origin to '*' — matches the upstream YT IFrame API's own behavior. Play/pause payloads have no sensitive data so wildcard is acceptable.
  • React Strict Mode double-wrap. Dev-mode Strict Mode invokes effects twice: mount → cleanup → mount. Each invocation was calling new YT.Player(iframe, ...), leaving two wrappers bound to the same iframe. YouTube's postMessage handshake only reaches the first wrapper, so the second (which ytPlayerRef ended up pointing at) never received onReady. Result: video played but getCurrentTime() returned 0 forever and the progress slider was stuck.
  • Added a wrapperCreatedRef ref that gates wrapper creation to exactly once per component instance. Second invocation only installs fresh timers. Also stopped calling YT.Player.destroy() in cleanup — it removes the iframe from the DOM out from under React, breaking the subsequent effect invocation. React handles iframe removal on real unmount.
  • Added a 500ms poll of getCurrentTime() and a pre-check inside the autoplay-timeout handler: if time is advancing, the tap-to-play overlay is auto-cleared or suppressed. Ground-truth fallback for any future case where the API handshake silently fails.
  • Overlay click now clears blocked optimistically. Previous behavior waited for onStateChange confirmation — which sometimes never arrives — and left the user staring at a black overlay on top of a playing video.

Misleading "Connecting to Spotify…" in YouTube mode

  • Landing on the player in YouTube-only mode still showed "Connecting to Spotify…" until the first YouTube track resolved — the loading message was keyed on !deviceId, which is forever-null when the Spotify SDK is deliberately not initialized.
  • Gated the message on source. Also confirmed via a [spotify-sdk] skipping init — youtubeOnly=true log that the SDK really does not run in YouTube mode.

2026-04-01

Problem: Spotify rate limiting — stats show nothing

  • Hitting Spotify's 30-request/30s rate limit repeatedly. But the in-app call tracker showed zero calls at the time of the 429.
  • Root cause: the call tracker only measures calls through /api/next-song. All browser-side Spotify calls — SDK internal retries, polling, device reclaim — are invisible to it.
  • Three sources identified:
    1. SDK auth retry loop: on authentication_error the SDK kept retrying _refreshToken → GET /v1/melody/v1/check_scope in a tight loop.
    2. Polling: getCurrentState() every 1 second — 60 calls/minute direct to Spotify.
    3. visibilitychange reclaim: fired PUT /v1/me/player on every tab switch with no throttle.

Fix: Stop SDK retry loop

  • In authentication_error listener: call player.disconnect() before setting error state. This stops the SDK from spinning on failed token refresh.

Fix: Replace Spotify polling with local animation

  • Removed getCurrentState() polling entirely — was the largest source of invisible Spotify calls.
  • Replaced with a 250ms local setInterval that increments sliderRef by 250ms when not paused and not seeking. Zero Spotify API calls.
  • Auto-advance (fire at 98% of duration) moved into the local timer. Progress bar is smooth; player_state_changed events resync position on real state changes.
  • visibilitychange handler throttled to at most one reclaim every 10 seconds.

Fix: Crossfade between tracks

  • Added fadeVolume(player, from, to): 20 steps over 700ms using player.setVolume() (SDK-local call, not a Spotify Web API call — no rate limit concern).
  • advanceWithFade(): fades out current track, sets pendingFadeInRef, calls advance().
  • Play effect: when pendingFadeInRef is set, starts new track at volume 0 then fades in.

Fix: Rating hammering Spotify

  • Every rating triggered a full LLM + Spotify lookup cycle, hammering the rate limit on rapid slider adjustments.
  • Split into two phases:
    1. On rating: fetchProfileOnly() — calls /api/next-song with profileOnly: true. LLM runs, Spotify lookup skipped. Returns raw song suggestions + updated profile. Suggestions shown in "DJ is thinking…" panel below the player.
    2. When songs actually needed: resolvePending() — calls /api/next-song with songsToResolve: [...]. Skips LLM entirely, just resolves the suggestions to Spotify tracks. Triggered lazily by the fill effect when buffer is empty.

Fix: Profile text quality

  • The "What I know about you" profile was a structured coordinate log (LIKED: (42,28) warm soul | DISLIKED: ...) — more like a debug dump than a human description.
  • Updated the LLM system prompt: profile field is now 2-3 natural sentences describing the listener's taste in terms of genres, eras, moods, instruments, and energy. Under 60 words. "Write like a DJ's notes on their audience."

2026-03-30

Problem: 401 Unauthorized spam on Vercel

  • On Vercel, POST /api/next-song immediately returned 401 (Unauthorized).
  • The client had no backoff for this error — the auto-fill effect loop retried immediately and endlessly (hundreds of requests per session).
  • User saw "Could not load songs. Try again." with the browser network tab flooding with 401s.
  • Root cause #1: Infinite retry loop — any non-rate-limited error from fetchToBuffer set loadingQueue=false in .finally(), which re-triggered the auto-fill useEffect, which called fetchToBuffer() again. No backoff for generic errors.

Fix: Infinite retry loop

  • Added class AuthError extends Error for 401 responses specifically.
  • In fetchCards: throw AuthError when res.status === 401.
  • In fetchToBuffer catch: on AuthError, redirect to / (re-auth). On generic errors, apply the same 30-second backoff used for rate limiting.

Problem: Refresh token cookie missing on Vercel

  • The 401 from /api/next-song was caused by missing cookies: access token and/or refresh token not found server-side.
  • Vercel log: "refreshSpotifyAccessToken: no refresh token cookie found".
  • The user was on /player (SSR reads the access token cookie fine), but the API route couldn't find it.
  • Root cause #2: The OAuth callback (app/callback/route.ts) used cookies().set() from next/headers followed by redirect() from next/navigation. In Next.js 16, redirect() throws a special error — cookies set before the throw may not be flushed into the response headers. On Vercel this meant only the first cookie (access token) was reliably set; the expiry and refresh token cookies were dropped.

Attempts to fix cookie setting

  • Tried NextResponse.redirect() with response.cookies.set() — broke local login entirely.
  • Reverted; added logging to confirm Spotify returns a refresh token (it does).
  • Tried plain new Response(null, { status: 302, headers }) with explicit Set-Cookie headers — worked locally, still failed on Vercel.
  • Current attempt: return a 200 HTML response with Set-Cookie headers and window.location.replace('/player'). Avoids redirect entirely — Vercel's edge network may strip Set-Cookie from 3xx responses but should leave them on 200 responses.

Why it works locally but not on Vercel

  • Locally (http://localhost:3000): NODE_ENV=development, cookies have no Secure flag, responses flow directly from Next.js dev server to browser with no proxy.
  • On Vercel: NODE_ENV=production, cookies have Secure; SameSite=Lax. All traffic goes through Vercel's edge network proxy. The proxy may strip Set-Cookie headers from 302 redirect responses (either for security, caching, or CDN normalisation reasons). The 200 HTML approach bypasses this.

Status

  • Infinite retry loop: fixed (30-second backoff on all errors; immediate redirect to login on 401).
  • Cookie persistence on Vercel: in progress — testing 200 HTML response approach.
